# Keeping the tutor in the target language

- The tutor must always reply in the **target language** (the one being learned), even when the learner types/speaks in their native language. This is enforced two ways and both are needed:
  1. A strong system prompt at scan time with explicit "CRITICAL LANGUAGE RULES".
  2. A **high-recency `system` reminder pushed AFTER the latest user message** on every turn in the streaming message route. Without this, long chats drift back to English once the learner replies in their native language.
- **A conversation's language is INTRINSIC, not the global app setting.** It is fixed at creation (scan/free chat); the whole message history + original system prompt are written in it. The server anchors every turn (and grading) to the conversation's OWN language: `target_language`/`native_language` columns first, then a title-segment parse, then the request value only as a legacy-row fallback. The server must NEVER overwrite the stored language from the request.
  - **Why:** the old code prioritized the request's `targetLanguage` (global app setting) and re-persisted it. When you opened a chat whose history was in language Y while the global was X, the late "reply in X" reminder fought the Y system prompt + Y history and the model **blended both languages**. Anchoring to the conversation's own language guarantees the reminder always matches the history.
  - **How to apply:** in `conversations.ts`, resolve `targetLanguage`/`nativeLanguage` as `safeLanguage(owned.col) || safeLanguage(title[1]) || requestValue`. Do not add any `db.update(...).set({ targetLanguage })` re-persist on the message route. Validate every language with `safeLanguage()` before it reaches a prompt.
- **Opening a past chat must never touch `nativeLanguage`.** `nativeLanguage` drives the app UI locale (`useT` → `translate(locale)`). The History switch-and-open may update the global `targetLanguage` (so the home "Learning X" indicator + new scans follow what the user chose) but must not swap `nativeLanguage`.
  - **Why:** a user learning Russian tapped a locked English chat; the old swap (`if (lang === nativeLanguage) update("nativeLanguage", targetLanguage)`) flipped native→Russian, turning the entire UI Russian ("Изучаю English").
- **Title parse is only a fallback.** Do NOT primarily re-derive language by parsing the title (`itemName • targetLanguage`) — item names can contain the ` • ` separator. It sits between the stored column and the request value as a self-heal for legacy/corrupted rows.
- **History is grouped by language; the lock is now passive.** Conversations are grouped into `SectionList` sections by their language (parsed from title, resolved case-insensitively to a known `LANGUAGES` member). A chat whose language differs from `prefs.targetLanguage` is shown locked (dimmed + lock icon) and tapping prompts "switch to X language" (switch-and-open) — but switching only changes the global `targetLanguage` and opens; the conversation still runs in its own language regardless. Unknown/unparseable title languages open directly, never locked. The lock UI and the open-gate MUST use the same resolver.
- **No inline translations in tutor replies.** The conversational tutor must reply in the **target language only** — never append a native-language gloss/translation in parentheses. Translation is on-demand: the client has a Translate button backed by `POST /openai/translate`. This rule must be repeated in EVERY prompt layer or it leaks back in: scan-time system prompt (scan.ts), free-chat system prompt + per-turn reminder (conversations.ts), `difficultyInstructions`/`difficultyReminder` (difficulty.ts, all 3 levels), and `speakingStyleRules` (languages.ts). Word Bank / Sentence Bank generators (vocab.ts/sentences.ts) DO emit a `translation` field — that's a separate card feature, leave it.
  - **Why:** users saw the tutor translate its own sentence before tapping Translate; older prompts explicitly asked for parenthetical native-language glosses at beginner level.
- **Vision-model item labels are untrusted** and get interpolated into the high-priority system prompt; sanitize them (collapse whitespace, cap length) before interpolation to avoid prompt-injection via odd image captions.
  - **Why:** an architect review flagged both the brittle title parse and the injection surface.
- **The `SUPPORTED_LANGUAGES` allowlist must be enforced at EVERY entry point that interpolates a language into a prompt, AND on every stored-value fallback** — not just on the request value. Languages are validated on the message/grade requests but also persisted on `conversations` and reused as a fallback; if any write path (e.g. `/scan`) skips validation, an unsafe stored value flows into a later prompt (grading). Validate languages where they enter (reject 400) and wrap stored fallbacks in `safeLanguage()`. The allowlist + `safeLanguage()` live in `artifacts/api-server/src/lib/languages.ts` (shared by scan + conversations routes).
  - **Why:** an architect review found `/scan` persisted unvalidated languages that the grade route then interpolated, defeating the allowlist.
